# Feature Definitions (calib_optionA_minimal_oofspan.csv)

This README documents how each column in `calib_optionA_minimal_oofspan.csv` was produced for the Option-A probability calibration / trading-rule work.

## Time index
- **Date**: Trading day *t* (the day you form the decision). All features must be known by your decision cutoff on Date *t*. The label/return is for *t → t+1*.

## Base signal and derived confidence
- **oof_prediction**: Out-of-fold (OOF) prediction from the base model for next-day return/direction. Each row's prediction was generated by a base model that was **not trained** on that row's day.

- **pred_z**: Causal z-score of `oof_prediction` relative to its own recent history.
  - Window: 252 trading days (approx 1 year)
  - Computation (causal):
    - `mu_t = mean(oof_prediction_{t-252 .. t-1})`
    - `sd_t = std(oof_prediction_{t-252 .. t-1})` (population std, ddof=0)
    - `pred_z_t = (oof_prediction_t - mu_t) / sd_t`
  - Implementation detail: rolling mean/std are **shifted by 1 day**, so the current day's prediction is never included in its own normalization.
  - Note: early dates are dropped until enough history exists (min_periods=80 in the build script).

- **abs_pred_z**: Absolute value of the confidence score.
  - `abs_pred_z_t = abs(pred_z_t)`

## Realized next-day outcome
- **y_next_log_return**: Realized next-day log return for XAUUSD (or the gold instrument you modeled), aligned so that it represents the move from Date *t* to Date *t+1*.

## Supervision targets (labels)
- **target_up**: Next-day direction label.
  - `target_up_t = 1 if y_next_log_return_t > 0 else 0`

- **target_correct**: Whether the base model's direction was correct.
  - `pred_dir_t = sign(oof_prediction_t)`  (positive → up, negative → down, zero → flat)
  - `real_dir_t = sign(y_next_log_return_t)`
  - `target_correct_t = 1 if pred_dir_t == real_dir_t else 0`
  - Note: if `y_next_log_return_t == 0`, then `real_dir_t = 0` and `target_correct_t` is 1 only if `oof_prediction_t == 0` (rare).

## Regime/context features (sourced from your dataset)
These two columns were **copied** from your `dataset.csv` by matching on `Date`. Their internal formulas are whatever you used during feature engineering/training; this file does not re-derive them.

- **Macro_Fast**: A scalar macro/regime indicator for Date *t* (must be computed causally at the same decision cutoff as the rest of your features).
  - Source: `dataset.csv` column `Macro_Fast` merged on `Date`.

- **Market_State**: A categorical regime label for Date *t* (e.g., `sideways_neutral`, etc.).
  - Source: `dataset.csv` column `Market_State` merged on `Date`.
  - When training models, this is typically one-hot encoded with `handle_unknown='ignore'` for safety.

## Build provenance
This file was built by:
1) Taking OOF rows where `has_prediction == True` and `oof_prediction`/`y_next_log_return` were non-null.
2) Computing `pred_z`/`abs_pred_z` from the OOF prediction stream using a trailing 252-day window with a 1-day shift (past-only).
3) Creating `target_up` and `target_correct` from `y_next_log_return` and `oof_prediction`.
4) Merging `Macro_Fast` and `Market_State` from `dataset.csv` by `Date`.
5) Dropping rows without a valid `pred_z` (warmup period).

## What you should keep invariant in production
- The **timestamp convention**: Date *t* features must be available by your decision time on Date *t*; labels use Date *t+1*.
- The **pred_z calculation**: use only historical predictions up to *t-1* (shifted rolling).
- The exact definitions of `Macro_Fast` and `Market_State` used in training (same code, same cutoffs, same calendars).
